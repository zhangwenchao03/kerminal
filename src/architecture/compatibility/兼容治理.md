# 兼容清单与退役治理

## 目标

`registry.json` 是 Kerminal 当前兼容路径的唯一清单真相源。清单只增加可观测性和退役门禁，不决定业务主路径，也不能作为恢复已删除兼容实现的依据。

Rust 入口位于 `src-tauri/src/services/compatibility_registry.rs`，TypeScript 入口位于 `compatibilityRegistry.ts`。两端都从同一 JSON manifest 读取稳定 ID、分类、生命周期、owner、允许原因和退役策略。

## 分类

| 分类 | 含义 |
| --- | --- |
| `runtime-fallback` | 主运行时能力不可用时的受控降级，例如 GPU、Managed SSH、SFTP 或配置 watcher |
| `runtime-patch` | 针对已知运行时兼容问题的局部补丁，例如 Tauri WebView 中的 xterm 补丁 |
| `schema-compatibility` | 读取旧数据或交换格式，例如 Workspace V1 和 Snippet V1 |
| `semantic-compatibility` | 保留旧调用语义，例如命令历史空范围表示全量清理 |
| `preview-adapter` | 非 Tauri 浏览器预览使用的显式运行模式 |
| `startup-recovery` | 启动阶段针对已知瞬态错误的有界恢复 |
| `diagnostic-policy` | 约束静默异常和兼容指标的治理规则 |

`sunset` 表示可以进入退役流程；`supported-mode` 是仍需长期支持的显式模式；`governance` 是长期规则。长期支持项不得伪造退役任务。

## 现有兼容点

| 稳定 ID | 实现语义 | 归一化启用原因 |
| --- | --- | --- |
| `terminal.gpu-fallback` | GPU 上下文丢失、atlas 清理失败、恢复风暴或软件 GPU 时降级 CPU | `context-lost`、`atlas-clear-failed`、`recovery-storm`、`software-gpu` |
| `terminal.xterm-webview-patch` | Tauri WebView 启动前修正 xterm 依赖兼容 | `tauri-webview` |
| `managed-ssh.legacy-fallback` | 仅 backend 未接线或 capability 明确不支持时允许旧路径 | `backend-unwired`、`backend-unsupported`、`capability-unsupported` |
| `sftp.transfer-polling` | Tauri event 不可用时轮询，event 健康和后台状态下执行低频安全轮询 | `event-channel-unavailable`、`event-channel-healthy-safety-poll`、`document-hidden` |
| `config-watcher.polling` | native watcher 不可用或测试/诊断显式选择 polling | `native-watcher-unavailable`、`explicit-polling` |
| `workspace.schema-v1-migration` | 读取 V1 或未版本化 workspace session 并规范化为当前模型 | `schema-v1`、`unversioned-session` |
| `runtime.browser-preview` | 浏览器中使用显式 preview adapter，不调用 Tauri IPC | `outside-tauri` |
| `startup.dynamic-import-retry` | 开发模式动态导入抓取失败或 Vite optimize dep 过期时有界重试 | `transient-import-fetch`、`outdated-optimize-dep` |
| `snippet.schema-v1` | Snippet 导入导出继续接受 V1 交换格式 | `schema-v1` |
| `command-history.empty-scope-clear` | 清理范围为空时保留全量清空旧语义 | `empty-scope` |
| `diagnostics.silent-catch-policy` | 静默 catch 必须选择聚合计数、明确 best-effort 或用户可见处置 | `aggregate-counter`、`best-effort-ignore`、`user-visible` |

Managed SSH 现有 `managed-*-unwired` / `managed-*-unsupported` marker 接入 registry 时应归一化为上述稳定原因。认证失败、用户取消、host key 变化、连接超时、远端非零退出和 channel/subsystem 打开失败不得映射成兼容降级。

## 指标与脱敏

公开快照只允许输出：稳定 ID、分类、生命周期、activation count 和 failure count。以下内容禁止进入兼容指标、runtime snapshot 或 tool help：

- 异常正文和 stderr；
- 文件系统路径、主机地址、用户名和 host identity；
- 命令、文件内容、凭据、token、私钥或其它 secret；
- manifest 的 implementation refs、owner、退役日期和内部任务细节。

TypeScript 指标输入即使携带调用方 labels，快照构建器也会全部丢弃；Rust 指标类型不提供 detail/labels 字段。

## 静默异常策略

每个静默 `catch` 必须选择且只选择一种处置：

1. `aggregate-counter`：增加有界聚合计数，reason 使用 registry 稳定枚举。
2. `best-effort-ignore`：功能明确为辅助能力，失败不会改变主流程正确性，并用代码注释说明边界。
3. `user-visible`：失败影响用户目标，转换成脱敏的用户可见诊断。

不得记录原始异常、路径或 secret，也不得用静默 catch 把认证、权限、数据损坏或主流程启动失败伪装成兼容成功。

## 退役门禁

只有 `sunset` 项可申请退役。自动门禁同时要求：

- 连续零观测窗口达到 manifest 的 `minimumZeroWindows`；
- 兼容路径及相邻主路径回归测试全部通过；
- 回滚步骤已经形成可执行记录。

门禁通过只表示具备评审条件，不直接删除代码。删除仍需由对应 `targetTask` 修改主路径、验证启动和更新 runtime snapshot/tool help 后完成。`supported-mode` 与 `governance` 项返回 `NotEligible`，不能通过伪造零计数删除。

## 主分支接线建议

主 lane 合并时只做以下最小接线：

1. 在 `kerminal.runtime_snapshot` 增加 `compatibility` 聚合快照，不返回完整 manifest。
2. 在 `kerminal.tool_help` 说明兼容指标字段和脱敏限制，不暴露源码路径与内部退役信息。
3. 现有 GPU、Managed SSH、SFTP 和 watcher 诊断点只上报稳定 ID、归一化 reason 与计数；不得改变当前 fallback 判断。
4. 前端 browser preview、startup retry、workspace/snippet migration 与静默 catch 分批接入计数，不在 TASK-013 批量修改现有调用链。
