# 并行 Agent 工作流调研记录

日期：2026-06-21

## 调研问题

当前 Kerminal 有多个 Codex 会话同时跑：一个做代码规范/架构优化，一个做 AI 助手逻辑。目标不是强制串行，而是让多个任务可以同时推进，并且在修改同一文件时互相可见、可复核、可继续。

## 外部依据

- Git worktree 是 Git 官方支持的多工作树能力：同一仓库可以有多个 working tree，用于并行检出不同分支或 detached HEAD。来源：[git-worktree](https://git-scm.com/docs/git-worktree)。
- Codex app 官方工作流也把 worktree 作为并行任务隔离方式：多个独立任务可以在同一项目里运行且不互相干扰；Git 仓库的后台任务推荐使用 dedicated background worktree。来源：[Codex app Worktrees](https://developers.openai.com/codex/app/worktrees)、[Codex app Automations](https://developers.openai.com/codex/app/automations)。
- Codex worktree 文档特别指出 ignored 文件不会自动随 Git worktree 移动，除非用 `.worktreeinclude` 或其他机制处理。这个点解释了本项目 `.codex/.updeng/AGENTS.md` 被 ignore 后，在并行 lane 状态里容易“隐身”。来源：[Codex app Worktrees](https://developers.openai.com/codex/app/worktrees)。
- Git rerere 能记录并复用已经人工解决过的冲突，适合长期 topic branch 反复遇到同类冲突的场景，但仍需要最终 diff/sanity check。来源：[git-rerere](https://git-scm.com/docs/git-rerere)。
- Git attributes 支持 built-in 和 custom merge driver；其中 union merge 会保留双方行但顺序可能不可靠，官方也要求使用者理解影响并复核结果。来源：[gitattributes](https://git-scm.com/docs/gitattributes)。
- OpenAI Agent Improvement Loop 的官方 cookbook 把 traces、human feedback、evals、ranked recommendations 和 Codex handoff 连成闭环；关键不是“模型自动记住”，而是把反馈和失败转成可复跑、可审查的改进证据。来源：[Build an Agent Improvement Loop with Traces, Evals, and Codex](https://developers.openai.com/cookbook/examples/agents_sdk/agent_improvement_loop)。
- Reflexion 把执行反馈、失败信号和语言化反思写入 agent 后续决策记忆，用于减少重复错误。来源：[Reflexion: Language Agents with Verbal Reinforcement Learning](https://arxiv.org/abs/2303.11366)。
- Self-Refine 使用模型自身反馈迭代改进输出，适合把“回答/实现质量问题”拆成 feedback 与 refine 两步，而不是一次性相信首答。来源：[Self-Refine: Iterative Refinement with Self-Feedback](https://arxiv.org/abs/2303.17651)。
- Voyager 通过执行错误、环境反馈、自验证和技能库沉淀让 agent 持续积累可复用能力；对本项目的启发是把工具失败和验证失败纳入候选队列，而不只记录用户纠正。来源：[Voyager: An Open-Ended Embodied Agent with Large Language Models](https://arxiv.org/abs/2305.16291)。

## 采用结论

1. 新 planned/high-risk/evolution 任务默认走独立 worktree/分支；当前历史脏主工作区只作为例外继续。
2. 同一文件并行修改不默认阻塞，采用 warn-and-sync：hook 提醒、agent 读 status/计划/diff/checkpoint、最小兼容修改。
3. 每个 lane 必须有可复核 checkpoint：tracked 文件用 patch；untracked/ignored 工作流资产用文本 snapshot；大文件只记录 hash 和省略原因。
4. `status.md` 是所有会话的同步入口，必须列出 active lanes、共享路径、未归属路径和最近 checkpoint。
5. 自进化不直接改正式 skill；UserPromptSubmit 捕获重复反馈/工作流缺口，PostToolUse/Stop 捕获 AI 工具失败、验证失败和收口缺口，统一写入 `evolution-candidates.jsonl`，再由人工或后续计划采纳。

## 暂不采用

- 不启用全局 `git rerere.enabled`：这会改变仓库/用户的 Git 行为，当前先记录为后续可选增强。
- 不给共享代码文件配置 union merge driver：共享 Rust/TypeScript 注册文件需要语义复核，union merge 可能保留双方行但生成错误顺序。
- 不让 hook 自动合并或自动采纳 skill 变更：当前目标是生产级可见性和候选采集，不是无人监督地修改工作流规则。

## 落地映射

- `lanes.json`：登记工作拆分和所有权。
- `status.md`：每轮同步入口。
- `checkpoints/<lane-id>.json|patch|untracked.json`：对方可读取的改动证据。
- `pre-tool-use.cjs`：共享路径 warn-and-sync。
- `workflow-event.cjs`：工具事件后刷新 lane status。
- `skill-forced-eval.cjs`：每轮注入 lane 状态，并捕获自进化候选。
