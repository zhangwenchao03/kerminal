# 代码规范、设计模式与性能治理实施计划

## 目标

- 建立 Kerminal 的代码规范门禁、架构重构边界和性能量测基线。
- 在现有代码上采用适合的设计模式，优先解决真实耦合、测试困难、协议漂移和性能热点。
- 用可验证切片逐步迁移，不影响当前 active 的 UI 调整工作。

## 非目标

- 不在一轮内重写全仓架构。
- 不强行落地全部 23 种 GoF 设计模式。
- 不改变用户可见行为，除非切片验收明确要求。
- 不在没有 profiling 或测试基线前宣称性能优化完成。

## 实施原则

- 稳定 facade，内部深化：对外入口尽量不变，先抽私有深模块。
- 一次只迁移一个行为切片，迁移前补测试或确认已有测试。
- 设计模式只服务于解耦、替换、测试或性能，不把模式名作为目标。
- 性能优化必须先有量测基线，再做修改。
- 前端 UI 相关改动必须验证浅色、深色和跟随系统主题。

## 风险

- **过度抽象**：用模式替代真实问题。对策：每个抽象必须写清不变量、调用方收益和测试改善。
- **大规模冲突**：当前已有 active UI 调整和大量未提交改动。对策：本计划排队执行，实施时只碰本切片文件。
- **行为回归**：重构入口广。对策：先补窄测，再迁移一个 tracer bullet。
- **性能误判**：没有数据就优化。对策：先建立 baseline，再做定点修改。
- **启动回归**：Tauri/xterm/WebView 兼容敏感。对策：遵守 `freezePrototype=false` 规则，涉及启动路径必须跑真实 smoke。
