# 垂直切片规划参考

## 切片定义

一个垂直切片交付一条可观察、可验证的端到端行为。它可以很窄，但不能只完成某一层。

## 好切片

- 用户或系统能独立验证。
- 覆盖必要的数据、业务、接口、UI、权限或错误路径。
- 有明确验收标准和验证命令。
- 依赖关系少，能按顺序交付。

## 坏切片

- “建表”“写接口”“做页面”这种横向层任务。
- 完成后不能 demo 或验证。
- 验收标准只是“代码完成”。
- 需要大量未确认设计才能开始，却标成 AFK。

## AFK/HITL 判断

AFK：

- 范围明确。
- 验收可自动或本地验证。
- 不需要业务取舍、生产权限或 UI 审美判断。
- 失败时有明确回滚或重试路径。

HITL：

- 需要人工确认交互、文案、权限、合规或上线窗口。
- 依赖外部账号、生产数据、客户反馈。
- 有多个合理方案且取舍不是纯技术问题。

## Issue 模板

```markdown
## What to build

<端到端行为，不按层描述>

## Acceptance criteria

- [ ] <可观察结果>
- [ ] <错误/边界行为>
- [ ] <验证命令或人工验收>

## Blocked by

None

## Verification

- <命令或步骤>
```

## Agent Brief 模板

```markdown
## Context

<领域词汇、相关文档和约束>

## Task

<本切片要完成的行为>

## Boundaries

- In scope:
- Out of scope:

## Verification

- <命令>

## Stop conditions

- <遇到什么必须回到人类>
```
