# 技术决策记录

## 使用原则

- 只有影响长期维护、公共契约、架构边界、依赖、数据、安全、发布或团队协作的选择才写 ADR。
- ADR 记录最终决策和关键取舍，不记录冗长调研过程。
- 决策必须能被后续验证或回滚；不能只写“采用某方案”。
- 同一主题已有 ADR 时，新增 ADR 应标注“替代/修订”关系，不直接改写历史。

## 触发判断

需要 ADR 的典型情况：

- 选框架、数据库、消息队列、缓存、GIS、前端状态管理、构建/发布方案。
- 修改 API、数据模型、权限模型、部署拓扑或更新机制。
- 引入高影响依赖或替换基础设施。
- 选择性能、安全、运行约束和开发效率之间的取舍。

不需要 ADR 的情况：

- 单个小 bug 修复。
- 局部实现细节，且不会影响其他模块。
- 已有规范明确规定的选择。

## ADR 模板

```markdown
# ADR-<编号>: <决策标题>

## 状态
Proposed | Accepted | Superseded | Retired

## 背景
- <业务目标、问题、约束和触发原因>

## 决策驱动因素
- <可维护性、性能、安全、成本、交付时间、运行约束等>

## 备选方案

| 方案 | 优点 | 缺点 | 风险 | 验证方式 |
| --- | --- | --- | --- | --- |
| A |  |  |  |  |
| B |  |  |  |  |

## 决策
采用 <方案>。

原因：
- <关键理由>

## 影响
- 正向影响：<...>
- 负向影响：<...>
- 需要同步修改：<代码、配置、文档、测试、发布>

## 回滚或替代
- <如果失败，如何撤回或切换>

## 验证
- <命令、指标、试点范围或验收标准>
```

## 选型比较维度

| 维度 | 需要回答的问题 |
|------|----------------|
| 运行约束 | 是否符合当前运行时、平台、部署环境和团队工具链 |
| 可维护性 | 是否降低复杂度，是否容易测试和定位问题 |
| 安全性 | 是否扩大权限、密钥、网络或数据泄露面 |
| 性能 | 是否影响启动、内存、吞吐、包体积或构建时间 |
| 成本 | 是否引入许可证、云资源、运维或学习成本 |
| 可替换性 | 是否锁定供应商或深度侵入业务代码 |
| 生态成熟度 | 是否维护活跃、文档完整、社区问题可搜索 |

## 文档位置

优先使用项目已有约定：

- `.updeng/docs/decisions/ADR-xxxx-title.md`
- `docs/adr/ADR-xxxx-title.md`

新增 ADR 后同步索引文件，如 `.updeng/docs/decisions/README.md` 或 `docs/adr/README.md`。

## 常见错误

| 错误做法 | 正确做法 |
|---------|---------|
| 只写结论不写约束 | 写清背景、驱动因素和替代方案 |
| 把调研笔记原样塞进 ADR | 提炼决策相关证据 |
| 没有回滚方案 | 写明失败后的降级或替换路径 |
| 决策后不同步文档和测试 | 更新相关规范、README、验证命令 |
| 覆盖历史 ADR | 新建 Superseded ADR 并建立关联 |
