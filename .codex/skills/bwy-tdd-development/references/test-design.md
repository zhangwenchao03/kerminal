# 行为测试设计参考

## 好测试

- 从公开入口验证行为：API、CLI、UI、领域服务公开接口、Tauri command。
- 测试名使用领域语言，描述用户或系统可观察能力。
- 断言结果、状态变化、错误模式和副作用，而不是内部调用顺序。
- 尽量穿过真实协作路径，只替换不可控边界，例如外部支付、邮件、时钟、随机数。

## 坏测试

- 直接测试私有函数、内部 helper 或实现顺序。
- 大量 mock 内部模块，导致重构就失败。
- 只断言类型形状或快照，而不验证行为。
- 测试为当前实现量身定制，行为坏了仍然通过。
- 一口气写完所有测试，再按测试形状倒推实现。

## Mock 边界

可以 mock：

- 网络外部服务。
- 时间、随机数、文件系统危险写入。
- 付费或生产依赖。
- 难以稳定复现的系统边界。

避免 mock：

- 当前仓库内部业务模块之间的协作。
- 数据转换、权限判断、事务边界等真实风险点。
- 只为了让测试更容易而绕过真实入口。

## 测试命名

推荐格式：

```text
<actor or module> <does behavior> when <condition>
```

示例：

- `user can export filtered orders as xlsx`
- `auth rejects expired remote ticket`
- `generator keeps permission prefix for imported table`

## 选择测试层级

| 层级 | 使用时机 |
| --- | --- |
| 单元 | 纯算法、状态机、无外部依赖规则 |
| 集成 | 服务、数据库、权限、序列化、事务 |
| API/CLI | 用户或系统通过稳定入口调用 |
| E2E/UI | 关键页面流程、浏览器状态、真实交互 |

优先最高且稳定的入口。测试太低会耦合实现；测试太高会慢且难定位。选择能准确覆盖行为风险的最窄入口。
