# 诊断反馈回路参考

## 目标

反馈回路要让问题从“描述”变成“可重复信号”。优先追求快、稳定、症状准确。

## 回路类型

| 类型 | 适用场景 | 证据 |
| --- | --- | --- |
| 自动化测试 | 代码路径可在本地或 CI 触发 | 失败测试名、命令、断言 |
| HTTP/curl | API、网关、鉴权、序列化问题 | 请求、响应、状态码、日志 |
| CLI/脚本 | 批处理、生成器、迁移、文件输入 | 命令、fixture、stdout/stderr |
| 浏览器脚本 | UI、路由、渲染、网络交互 | DOM 断言、console、network trace |
| 回放 trace | 外部系统、复杂事件流 | HAR、payload、event log |
| 最小 harness | 服务启动重、依赖多、入口分散 | 临时代码路径、输入和输出 |
| 压力循环 | 偶发、并发、时序、缓存 | 循环次数、失败率、seed |
| 人工协助 | 需要真实账号、硬件、生产界面 | 操作步骤、观察记录、截图 |

## 质量标准

- 症状准确：失败内容等于用户报告的问题。
- 可重复：多次运行稳定，或偶发问题复现率足够高。
- 可收窄：能逐步减少输入、依赖或环境变量。
- 可验证修复：修复后同一回路能转绿。
- 可保留：能转成测试、脚本或文档；不能保留时说明原因。

## 偶发问题处理

- 固定时间、随机种子、时区、并发数、缓存目录和网络开关。
- 循环运行并记录失败率。
- 并行触发提高竞争窗口。
- 对比两个版本、配置或数据集。
- 必要时用 `git bisect run` 包装最小检查命令。

## 临时插桩规则

- 前缀统一：`[DEBUG-UPDENG-<短码>]`。
- 插桩放在能区分假设的边界上，不撒网式打印。
- 每条日志说明输入、关键状态和预测结果。
- 收口前用前缀全文搜索并删除。
