# 通用诊断工作流

## 使用边界

- 适用于任何语言和框架的缺陷、失败测试、错误日志、性能回退、间歇性问题和“现象不明”的排查。
- 本技能负责诊断纪律：反馈回路、复现、假设、证据、修复、回归、清理。
- 具体实现仍按专项技能执行：Java 用 `bwy-java-backend-development`，前端用 `bwy-frontend-development`，数据库用 `bwy-database-change-management`，远程生产用 `bwy-remote-ops-safety`。
- 如果用户只要求代码评审而不是排查已知现象，优先用 `bwy-code-review`。

## 核心原则

1. 没有反馈回路，不进入修复。
2. 先确认复现的是用户描述的问题，不修“旁边的另一个问题”。
3. 一次只验证一个可证伪假设。
4. 临时插桩必须可 grep、可删除、不中长期遗留。
5. 修复完成必须留下回归信号，或明确说明当前架构没有合适测试缝隙。

## 标准流程

### 1. 建立反馈回路

优先级从快到慢：

1. 自动化失败测试：单元、集成、端到端均可，优先能穿过真实调用链的最窄测试。
2. 命令或脚本：CLI、curl、HTTP fixture、日志回放、SQL dry-run。
3. 浏览器或 UI 自动化：Playwright、Puppeteer、DOM/console/network 断言。
4. 最小复现 harness：只启动必要模块，用固定输入触发问题。
5. 压力或随机循环：用于偶发、时序、并发、缓存和边界输入问题。
6. 人工协助回路：只有必须人工点击或访问外部环境时使用，并记录步骤、输入、观察结果。

反馈回路必须记录：

- 运行命令或操作入口。
- 期望结果和实际症状。
- 是否稳定复现；偶发问题记录复现率。
- 失败样本、请求、日志、截图或性能基线的位置。

更多构造方式见 [feedback-loops.md](references/feedback-loops.md)。

### 2. 复现并收窄

- 连续运行反馈回路，确认症状稳定或复现率足够高。
- 收窄输入、数据、配置、版本或环境差异，避免在完整系统里盲查。
- 如果无法复现，先列出已尝试的反馈回路和缺口，向用户要日志、HAR、录屏、fixture、环境访问或临时插桩授权。

### 3. 生成假设

在改代码前列出 3 到 5 个排序假设，每个假设必须可证伪：

```markdown
| 排名 | 假设 | 如果为真应观察到 | 验证方式 |
| --- | --- | --- | --- |
| 1 | <原因> | <预测> | <命令/插桩/对比> |
```

- 用户在线时，简短同步假设排序，接受领域信息修正。
- 用户不在线时，按最高置信度继续，但保留表格作为诊断证据。

### 4. 定点验证

- 每个 probe 对应一个假设预测。
- 优先使用调试器、REPL、断言、query plan、profile；其次才加日志。
- 临时日志统一前缀，例如 `[DEBUG-UPDENG-<短码>]`，收口时全文搜索并删除。
- 性能问题先基线测量，再比较；不要凭感觉改性能。

### 5. 修复和回归

- 找到原因后，先把最小复现转成回归测试或可重复验证脚本。
- 测试缝隙应覆盖真实问题模式；过浅测试只能作为辅助，不算回归闭环。
- 修复只覆盖已证实原因，避免顺手重构扩大变量。
- 修复后运行：最小回归信号、原始反馈回路、相关窄测试。

### 6. 清理和复盘

完成前必须检查：

- 原始问题不再复现。
- 回归测试或替代验证存在。
- 临时 debug 日志、harness、fixture 已删除或移到明确 debug 目录。
- 记录最终根因、修复点、验证命令和未覆盖风险。
- 如果没有合适测试缝隙，建议后续使用 `bwy-architecture-deepening` 改善模块 seam。

## 停止条件

- 无法建立可用反馈回路，且缺少用户或环境提供的复现证据。
- 需要访问生产、远程服务、真实数据或写操作时，切到 `bwy-remote-ops-safety` 或高风险流程。
- 诊断发现需求理解不一致时，回到 `bwy-domain-context` 或 `bwy-development-governance` 澄清目标。
