# 架构深化评估

## 使用边界

- 用于分析和规划架构改进，不默认直接大规模重构。
- 目标是提高局部性、杠杆、可测试性和 AI 可导航性。
- 已有 ADR 明确禁止的方向不要反复提出，除非现实摩擦足够大且需要重开决策。
- 具体实施仍走 `bwy-development-governance`；重大选择走 `bwy-tech-decision`。

## 术语

- **模块**：有接口和实现的单元，可以是函数、类、包、页面、服务或业务切片。
- **接口**：调用方必须知道的一切，包括类型、约束、错误、顺序、配置和副作用。
- **实现**：接口背后的内部细节。
- **深度**：小接口承载大能力的程度。
- **缝隙**：行为可以替换、测试或隔离的位置。
- **适配器**：在某个缝隙上满足接口的具体实现。
- **局部性**：修改、bug 和知识是否集中。
- **杠杆**：调用方通过少量接口获得多少稳定能力。

详细检查方法见 [deep-module-review.md](references/deep-module-review.md)。

## 工作流

### 1. 读取上下文

- 先读 `bwy-domain-context` 维护的领域词汇和相关 ADR。
- 读取最近似实现、调用链、测试和用户提到的痛点。
- 有 `.codegraph/` 时优先使用 CodeGraph 查看符号、调用者和文件结构。

### 2. 发现候选

寻找这些信号：

- 理解一个业务概念需要在很多薄文件间跳转。
- 模块接口几乎和实现一样复杂。
- 抽出的函数只方便测试，但真实 bug 发生在调用协作处。
- 多个调用方重复处理同一不变量、权限、事务、错误或数据形状。
- 测试只能测私有细节，无法通过稳定入口锁定行为。
- 新功能总要同时改一串无清晰拥有者的文件。

对每个候选做“删除测试”：如果删除该模块，复杂度是消失，还是分散回多个调用方。只包装一层、不增加局部性和杠杆的模块不是好候选。

### 3. 输出评估报告

对每个候选给出：

```markdown
## <候选名称>

- 文件/模块：
- 当前摩擦：
- 建议深化：
- 局部性收益：
- 杠杆收益：
- 测试改善：
- ADR 冲突：
- 推荐强度：Strong | Worth exploring | Speculative
```

需要图示时可写 Mermaid 或生成临时 HTML 报告，模板见 [report-template.md](references/report-template.md)。

### 4. 追问和定案

用户选择候选后：

- 澄清新模块承担的概念、接口、不变量和错误模式。
- 明确适配器数量；只有一个适配器时，判断是否是假缝隙。
- 设计可验证的迁移切片，而不是一次性重写。
- 新术语写入 `bwy-domain-context`；长期取舍写 ADR。

### 5. 交付实施计划

实施计划必须包含：

- 第一条可验证 tracer bullet。
- 入口替换和删除策略。
- 测试迁移顺序。
- 回滚方式。
- 每步可运行的验证命令。

## 禁止事项

- 不基于真实摩擦提出抽象。
- 不为了“整洁”拆分已经足够深的模块。
- 不在未锁定行为前重构核心路径。
- 不把架构报告直接当作实施完成证明。
