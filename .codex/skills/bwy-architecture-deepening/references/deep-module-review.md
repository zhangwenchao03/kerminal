# 深模块评估参考

## 评估维度

| 维度 | 问题 |
| --- | --- |
| 接口复杂度 | 调用方需要知道多少顺序、配置、错误和不变量 |
| 实现深度 | 模块内部是否承载足够行为，而不只是转发 |
| 局部性 | 变更是否集中在一个地方 |
| 杠杆 | 调用方是否因为接口稳定而少写代码 |
| 测试缝隙 | 是否能通过接口验证真实行为 |
| 适配器数量 | 是否存在至少两个真实实现或替换场景 |

## 常见浅模块

- 只改名不隐藏复杂度的 wrapper。
- 每个方法一行转调的 service。
- 为测试而抽出的纯函数，但调用方仍重复处理不变量。
- “utils” 聚合很多无关函数，没有领域边界。
- 前端组件只搬运 props，状态和副作用散在父级。

## 深化方式

- 把重复不变量推入模块接口背后。
- 用领域概念命名模块，而不是技术步骤命名。
- 将错误、权限、事务、缓存和序列化边界集中。
- 提供高层行为方法，减少调用方编排步骤。
- 把外部系统放在 adapter 后，业务代码依赖稳定接口。

## 删除测试

想象删除该模块：

- 如果复杂度消失，说明它可能只是多余间接层。
- 如果复杂度分散回多个调用方，说明它正在提供局部性。
- 如果删除后调用方更容易懂，说明当前接口可能比实现还复杂。

## 实施顺序

1. 先锁定当前行为测试。
2. 引入新接口或新模块。
3. 迁移一个调用方作为 tracer bullet。
4. 运行测试并对比行为。
5. 逐步迁移剩余调用方。
6. 删除被替换入口和重复逻辑。
