# 实现评审能力

## 评审姿态

优先输出问题，不写泛泛总结。重点找具体缺陷、行为回归、漏测、数据破坏风险、权限泄漏、竞态和文档漂移。不要把评审篇幅花在普通改动的夸奖上。

## 工作流程

1. 查看 `git diff --stat` 和变更文件。
2. 阅读受影响代码路径，而不只是补丁片段。
3. 先做需求符合性评审：对照需求、计划、接口契约、业务文档和任务清单，确认实现是否完成该做的事且没有改变不该变的行为。
4. 再做代码质量评审：检查正确性、安全性、事务、SQL、权限、异常、可维护性和性能风险。
5. 查看验证证据；能跑则跑。
6. 检查是否有范围漂移、无关重构、文档缺口或测试证据不足。
7. 按严重程度输出发现，并带文件和行号。
8. 没发现问题时明确说明，并列出剩余风险或未运行测试。

## 问题格式

```markdown
- [P1] <短标题> - <file>:<line>
  <为什么这是实际缺陷、何时触发、建议怎么改>
```

严重级别：

- `P0`：数据丢失、安全问题、无法构建、核心流程完全不可用。
- `P1`：高概率用户可见回归或业务结果错误。
- `P2`：边界缺陷、重要校验缺失、较快会咬人的可维护风险。
- `P3`：低风险优化；除非明确有价值，否则少提。

## 常用例子

评审起手命令：

```powershell
git status --short
git diff --stat
git diff -- <path>
```

缺陷写法：

```markdown
- [P1] 更新接口绕过数据权限 - src/main/java/.../ItemController.java:88
  当前接口直接按 id 更新记录，没有校验该记录是否属于调用人的数据范围。拥有 edit 权限但不拥有目标部门数据权限的用户可以修改不可见数据。建议在 Service 更新前复用同模块的 checkDataScope 或按 Mapper 数据权限查询确认记录可见。
```

无问题输出：

```markdown
未发现阻断性问题。已检查接口契约、权限、事务、SQL、异常和验证证据。
剩余风险：<未运行的测试/无法覆盖的环境/依赖外部系统的验证>
```

## 检查清单

- 行为变化必须有需求依据、影响范围和验证证据。
- 外部负载可空字段要注意包装类型/空值。
- list/page 接口保留分页和权限过滤。
- SQL 不因 join 意外改变计数。
- 前端 API 类型匹配后端字段名和可空性。
- 行为变化同步测试和文档。
- 差异只覆盖本次需求，不夹带风格化重构或未说明的接口变化。
- 验证证据能覆盖主路径和最高风险边界。
- 计划或任务清单中的每个验收点都有对应实现、测试或明确不做的说明。
- 后台管理类模块检查权限、日志、防重、分组校验、数据权限、导入导出、缓存失效、删除前校验是否跟同模块一致。
- 新增或修改前后端接口时，检查后端路由、权限标识、前端 API 路径、类型字段和日期范围参数是否对齐。
