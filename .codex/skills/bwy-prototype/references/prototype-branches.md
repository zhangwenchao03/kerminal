# 原型类型参考

## Logic 原型

用于验证状态机、业务规则、权限分支、队列/任务流。

要求：

- CLI 或脚本可一条命令运行。
- 内置 3 到 5 个关键场景。
- 每次操作打印完整关键状态。
- 不连接生产数据。

常见形式：

- `node scripts/prototype-*.mjs`
- `pnpm prototype:<name>`
- 临时测试 fixture 驱动的 reducer/state machine。

## UI 原型

用于比较页面布局、表单流程、交互密度、视觉方向。

要求：

- 使用现有前端栈和路由。
- 一个入口能切换多个方案。
- 变体使用同一数据集，便于比较。
- 不把说明性长文放到正式 UI。

常见切换方式：

- URL 参数：`?variant=a`
- 临时浮动切换条。
- Storybook 或项目已有 demo route。

## Spike 原型

用于验证库、协议、外部服务、性能或构建可行性。

要求：

- 最小依赖和最小配置。
- 打印成功条件、失败原因和限制。
- 记录引入成本、权限、许可证、部署影响。
- 成功后必须转 ADR 或计划；失败后记录不采纳原因。

## 收口矩阵

| 结果 | 动作 |
| --- | --- |
| 原型证明方案可行 | 删除原型并把决策写入计划/ADR |
| 原型代码可直接复用 | 吸收到正式实现，去掉 PROTOTYPE 标记并补测试 |
| 原型证明方案不可行 | 删除或归档结论，记录拒绝原因 |
| 仍需用户判断 | 保留最小原型并写清过期条件 |
