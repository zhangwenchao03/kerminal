# 原型探索

## 使用边界

- 原型是回答问题的临时代码，不是生产实现。
- 适用于需求尚不确定、状态模型难以纸面判断、UI 方向需要比较、技术集成风险高的场景。
- 不适用于已经明确的小改、必须直接修复的生产缺陷、或会误导为正式交付的半成品。

## 原型分支

先判断原型要回答的问题：

| 问题 | 原型类型 | 输出 |
| --- | --- | --- |
| 状态/业务逻辑是否成立 | logic | CLI、脚本、fixture 或小型交互程序 |
| UI/交互方向哪个好 | ui | 单一路由或页面中可切换的多个变体 |
| 技术可行性 | spike | 最小集成脚本、临时 adapter 或 isolated demo |

选择细节见 [prototype-branches.md](references/prototype-branches.md)。

## 规则

1. 在文件名、页面标题或注释中明确标记 `PROTOTYPE`。
2. 放在与目标模块接近的位置，便于理解上下文，但不能混进正式入口。
3. 提供一条运行命令。
4. 默认不持久化；如必须持久化，使用明确可擦除的本地文件或 scratch 数据库。
5. 暴露状态：每次动作后打印或渲染关键状态、输入、输出和错误。
6. 不追求完整错误处理、测试和抽象；只保留回答问题所需的代码。
7. 原型结束后必须删除、吸收到正式实现，或留下明确过期条件。

## 工作流

### 1. 定义问题

用一句话写清：

```markdown
原型问题：<这个原型要让我们学到什么>
非目标：<不验证什么>
成功信号：<用户或开发者看到什么就能判断>
删除/吸收条件：<何时删除或进入正式实现>
```

### 2. 选择承载方式

- 使用项目现有语言、路由、构建工具和测试 fixture。
- 不引入重量级依赖，除非原型的问题本身就是验证该依赖。
- UI 原型如果项目已有 dev server，使用现有路由体系；不要另起无关 demo app。

### 3. 构建最小可玩闭环

- Logic 原型：提供几个预置场景和自由输入方式。
- UI 原型：提供方案切换控件或 URL 参数，并保持同一数据集。
- Spike 原型：打印关键能力、失败原因、配置要求和限制。

### 4. 捕获结论

原型完成后，把结论写到最小拥有位置：

- 进入实现：写计划或 issue。
- 长期取舍：用 `bwy-tech-decision` 写 ADR。
- 领域词汇变化：用 `bwy-domain-context` 更新上下文。
- 不采纳：记录拒绝原因，避免后续重复建议。

### 5. 收口

交付时说明：

- 原型文件和运行命令。
- 回答了什么问题。
- 推荐采用、拒绝或继续探索的方案。
- 原型已删除、待删除，还是已被正式实现吸收。

## 风险控制

- 原型不得访问生产、真实写库或真实付费外部服务。
- 如果需要外部账号、密钥或生产数据，升级为高风险流程。
- 原型不能作为最终验证；正式实现仍需 `bwy-tdd-development` 或对应验证命令。
