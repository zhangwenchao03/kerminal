# 通用开发规范

## 使用原则

- 在任何项目代码修改前先确认该项目自身文档索引和 AGENTS.md；本技能只提供跨项目可复用的底线规范。
- 语言或框架专属规范同时适用时，先遵守该语言 skill，再用本技能补齐通用文档、注释、测试和对外边界要求。
- 发现本技能与项目现有规范冲突时，以当前项目更具体、更接近当前模块的约定为准。

## 文档治理

- 每个项目的文档入口、文档职责和开发流转以该项目自身文档索引为准。
- 同一描述只维护在一个稳定位置；其他文档只保留入口链接，不重复展开。
- 代码、配置、接口、运行方式或用户可见行为发生变化时，同步更新对应项目文档。
- 通用规范以本技能为准；修改规范时直接维护本技能。

## 注释与可读性

- 所有新增或重写的文件都需要在文件头或文件级注释中标注 `@author kongweiguang`；文件格式不支持注释、机器生成文件、锁文件、二进制资源除外。
- 关键逻辑、复杂流程和不易理解的约定需要补充必要注释，注释统一使用中文。
- 所有方法、属性、常量、枚举值、record 组件、配置 Bean 和公开扩展点都需要补充中文注释。
- 简单 getter/setter 可通过字段注释和类级说明覆盖，但不能让读者需要猜测字段含义、协议含义或生命周期边界。
- 注释要解释业务语义、协议边界、生命周期或复杂流程，不写“给变量赋值”这类重复代码字面含义的空注释。

## 文件规模与拆分

- 官方 React、Rust 和 Google 风格指南没有给出统一的“单文件最大行数”硬标准；本项目把文件规模作为团队可维护性门禁补充到规范中。
- 所有手写代码文件都必须控制在 1000 行以内，测试文件、测试夹具、脚本和测试辅助文件同样适用；锁文件、机器生成文件、vendor 代码和二进制资源除外。
- 单文件达到 800 行时视为预警，新功能不要继续堆入该文件；优先按职责拆出组件、Hook、领域模型、服务、fixture、断言 helper 或集成测试分组。
- 不能为了绕过行数限制把无关逻辑搬进“大杂烩 helper”；拆分后的文件仍需有清晰领域边界、单向依赖和可单独测试的行为。
- 交付前需要用仓库实际统计确认没有手写代码文件超过 1000 行；发现超限文件时，本次改动涉及范围内必须先拆分到合规。

## 测试要求

- 测试要尽量完整、覆盖全面，优先覆盖协议解析、异常输入、状态机边界、异步隔离、打断流程、配置默认值和公开扩展点。
- 新增功能必须同步补充能保护长期行为的测试。
- 测试需要按领域使用清晰的包名、类名、标签和展示名称标注，方便 IDE、Maven、Vitest、Cargo 或 CI 按类别查看与筛选。
- 如果确实无法补自动化测试，需要完成可复现的人工验证，并在交付说明中说明原因和验证方式。
- 涉及 UI、样式、窗口、WebView 或交互态时，必须运行真实应用或页面并截图对照；关键视觉差异未消除时，不把任务标记为完成。

## UI 运行截图验证

- Web/Tauri 前端页面：启动 dev server 或 `pnpm tauri dev`，用浏览器或 WebView 实际运行页面并截图。
- 原生桌面窗口（例如 Qt、Tauri 主窗口或系统对话框）：截运行窗口，不只看组件代码或静态文件。
- 有原型、设计图、参考 HTML、旧页面或用户截图时，必须并排比对布局分区、颜色、间距、字号字重、控件类型、文案和 hover/active/disabled/loading/empty/error 等状态。
- 改完后重新截图；如果关键差异仍在，验证结果视为失败，提交前继续修正。
- 在交付或 `verification.md` 中记录运行命令、窗口/页面 URL、截图路径、参考源和剩余差异。

## 设计与实现原则

- 方案选择优先考虑中长期可维护性、健壮性和扩展性，不为了短期通过验证牺牲清晰分层、协议约束、错误边界、测试覆盖和后续真实服务替换空间。
- 前端 React、后端 Java 或 Rust 实现功能时，如果已有成熟、维护良好的开源第三方库，优先评估并使用第三方库。
- 只有功能足够简单、引入依赖明显过重或会破坏现有分层时，才自行实现。
- 设计开发时优先采用成熟设计模式表达稳定变化点，例如 `Strategy`、`Factory`、`Observer/Hook`、`Facade`、`State`、`Adapter`、`Template Method`。
- 使用模式必须服务于解耦、替换、测试或扩展，不为了形式引入过度抽象。
- 修改代码时优先复用现有分层与扩展点；如果必须新增抽象，需要说明它解决的长期复杂度，避免为单个临时场景引入过度设计。

## 对外边界

- 对外协议、公开接口、配置项和 mock 行为要尽量稳定、类型清晰，并通过文档和测试固化边界，避免让调用方依赖隐式约定。
- 调整协议字段、配置项、启动方式或用户可见行为时，同步更新对应项目的 README、业务文档、完成态文档或规范文档。

## 交付检查

- 新增或重写文件是否已有 `@author kongweiguang` 标识。
- 公开类型、方法、属性、常量、配置和扩展点是否有中文注释。
- 复杂逻辑、协议边界、生命周期和错误边界是否用中文注释说明。
- 新增功能或缺陷修复是否有自动化测试；没有测试时是否记录人工验证。
- 对外协议、配置、运行方式或用户可见行为变化是否同步文档。
