# 需求发现与头脑风暴

## 使用边界

- 用于把模糊输入变成可执行方向，不直接编码。
- 本技能负责发现问题、提出候选、追问关键决策、收敛验收和路由下一步。
- 领域术语长期沉淀交给 `bwy-domain-context`；PRD/issue 拆分交给 `bwy-issue-planning`；原型交给 `bwy-prototype`；架构取舍交给 `bwy-architecture-deepening` 或 `bwy-tech-decision`；实现交给 `bwy-development-governance`。
- 如果用户明确说“直接实现”，且需求已经足够清楚，不强制进入长讨论。

## 核心原则

1. 先理解要解决的问题，再讨论功能形态。
2. 能通过代码和文档查到的问题先自己查，不把可发现事实推给用户。
3. 每次只问 1 个最高影响问题，并给出推荐答案。
4. 不用泛泛开放式问题拖慢用户；问题必须说明为什么重要。
5. 讨论结束必须产生下一步：不做、继续发现、写 PRD、做原型、做 ADR、拆 issue 或进入实现。

## 工作流

### 1. 快速定界

先用 4 句话以内复述：

- 用户想达成的结果。
- 当前不确定点。
- 可能影响的用户、流程、数据或系统边界。
- 你建议先澄清的第一件事。

如果任务涉及现有系统，先读 `AGENTS.md`、`README.md`、`.updeng/docs/` 和最近似代码；有 `.codegraph/` 时先用 CodeGraph。

### 2. 形成发现画布

维护一份临时发现画布，可以在对话中展示：

```markdown
## 发现画布

- 目标：
- 非目标：
- 用户/角色：
- 当前痛点：
- 成功信号：
- 约束：
- 备选方向：
- 待确认：
- 推荐下一步：
```

需要保存时写入 `.updeng/docs/discovery/YYYY-MM-DD-<topic>.md`，模板见 [discovery-note.md](references/discovery-note.md)。

### 3. 逐个追问

优先追问会改变方案形态的问题：

1. 目标用户是谁，何时遇到问题。
2. 成功标准是什么，什么算失败。
3. 本次必须做什么，明确不做什么。
4. 是否涉及数据、权限、审计、合规、生产或外部系统。
5. 需要自动化、人工确认，还是混合流程。
6. 需要快速验证方向，还是直接进入正式实现。

问题格式：

```markdown
问题：<一个具体问题>
推荐：<我的建议>
原因：<它会影响什么>
如果选择 A：<后果>
如果选择 B：<后果>
```

更多问题库见 [question-bank.md](references/question-bank.md)。

### 4. 给候选方案

当信息足够时给 2 到 4 个方案，不要只给单一路径：

```markdown
| 方案 | 适合场景 | 优点 | 代价/风险 | 验证方式 |
| --- | --- | --- | --- | --- |
```

对每个方案标注：

- 最小可行切片。
- 用户可见变化。
- 技术/数据/权限影响。
- 需要的验证或原型。

### 5. 收敛到下一步

用下面的决策树收口：

| 条件 | 下一步 |
| --- | --- |
| 术语或业务边界仍不稳定 | 使用 `bwy-domain-context` |
| 已明确产品范围，需要任务化 | 使用 `bwy-issue-planning` |
| 方向需要试用或比较 | 使用 `bwy-prototype` |
| 主要风险是架构/模块边界 | 使用 `bwy-architecture-deepening` |
| 是长期技术取舍 | 使用 `bwy-tech-decision` |
| 范围、验收和风险已清楚 | 使用 `bwy-development-governance` 进入实现 |

### 6. 保存发现结果

需要进入 plan、audit package 或长期有价值的讨论，写 `.updeng/docs/discovery/`：

- 记录最终目标、非目标、已确认决策、开放问题。
- 记录用户选择的方案和拒绝的方案。
- 链接后续 PRD、issue、原型、ADR 或计划。
- 不写聊天流水账。

## 停止条件

- 用户明确要求“先别问，直接按方案做”，且风险低、验收明确。
- 已经连续 3 个问题都不能改变方案，说明可以进入计划或实现。
- 缺少业务方信息且无法从代码查证，标记 blocked，列出所需输入。
